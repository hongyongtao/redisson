1. 不传 leaseTime 的锁，源码里到底怎么做？
以 RLock 的默认用法为例：
RLock lock = redissonClient.getLock("xxx");lock.lock();              // 不传 leaseTime// 或 tryLock() 不带 leaseTime 的重载
在 RedissonLock 里，大致是这样的流程（伪代码）：
lock() → 调用内部重载 lock(-1, null, false)，也就是 用 -1 表示“用户没指定 leaseTime”。
进入真正加锁逻辑时，会先算一个 internalLockLeaseTime：
   if (leaseTime == -1) {       internalLockLeaseTime = config.getLockWatchdogTimeout();   } else {       internalLockLeaseTime = leaseTime;   }
这里的 config.getLockWatchdogTimeout() 就是配置里的 lockWatchdogTimeout，默认 30_000 毫秒。
也就是说：你没传 leaseTime 时，内部会自动用 lockWatchdogTimeout 作为“初始过期时间”。
构造 Lua 脚本参数，执行 EVAL：
Lua 里会对锁 key 执行 SET + PEXPIRE internalLockLeaseTime，也就是 TTL = internalLockLeaseTime。
这一步就完成了你关心的那件事：即使你没传 leaseTime，锁 key 在 Redis 里一开始就是带 TTL 的。
所以答案是：
> 第二种“不传过期时间”的情况，默认也是有过期时间的，取的就是 lockWatchdogTimeout（默认 30s）
这个配置是redssion内嵌的默认配置，就算你在 redisson.yml / 代码 Config 里完全没提 lockWatchdogTimeout，它默认也是 30_000ms（30 秒

你真正需要注意的点
如果用“指定 leaseTime”的方式（不走看门狗），要保证 leaseTime ≥ 你业务的最大执行时间，否则锁可能提前释放引入并发问题。
如果用“默认 lock() + 看门狗”，要意识到：
进程大 GC 或长时间挂起，极端情况下仍可能发生：GC 导致 TTL 没续上 → 锁过期 → 其他节点拿到锁 → 业务逻辑等价于“临界区被拆成两次执行”，这是所有基于“超时释放”的分布式锁绕不开的理论风险，只能降低概率（调大 lockWatchdogTimeout 等）
