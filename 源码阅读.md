## 1. 项目整体结构总览

> 说明：以下基于 Redisson 官方开源仓库的典型结构整理，与你本地分支可能存在少量差异，但核心类与包结构基本一致。阅读时可以配合 IDE 的“跳转到定义/实现”逐步对照。

- **根模块**
  - `README.md`：项目简介与特性列表（你已经能看到）。
  - `pom.xml` / `build.gradle`：构建配置，声明核心模块，如：
    - `redisson`：核心客户端实现（最重要）。
    - `redisson-spring-boot-starter`：Spring Boot 集成。
    - `redisson-hibernate`、`redisson-mybatis` 等：与第三方框架集成。
- **核心源码包（以 Maven core 模块为例）**
  - `src/main/java/org/redisson/*`
    - `org.redisson`：`Redisson`、`RedissonClient` 及各类分布式对象的默认实现类。
    - `org.redisson.api`：对外暴露的接口层（`RMap`、`RLock` 等），业务代码只依赖这些接口即可。
    - `org.redisson.config`：配置对象及各种部署模式配置（单机、哨兵、集群、多主等）。
    - `org.redisson.connection`：连接管理器、拓扑发现、连接池封装。
    - `org.redisson.client`：基于 Netty 的 Redis 原生协议客户端、命令编解码、连接抽象。
    - `org.redisson.command`：命令执行服务，负责路由、重试、超时等。
    - `org.redisson.codec`：序列化/反序列化（Jackson、Kryo、JDK 等）。
    - `org.redisson.eviction` / `org.redisson.cache`：本地缓存与失效策略。
    - `org.redisson.jcache` / `org.redisson.spring` 等：各类集成实现。

阅读建议：**优先聚焦 `api`、`Redisson` 启动流程、`connection`、`command`、锁与集合几个包**，其它包可按需再深入。

## 2. 启动流程：从配置到 `RedissonClient`

### 2.1 对外入口：`RedissonClient` 与 `Redisson`

- **关键接口**
  - `org.redisson.api.RedissonClient`
    - Redisson 的“总控”接口，提供所有分布式对象工厂方法，如：
      - `getMap(String name)` → `RMap`
      - `getLock(String name)` → `RLock`
      - `getBucket(String name)` → `RBucket`
      - `getRateLimiter(String name)` → `RRateLimiter`
    - 内部一般由 `Redisson` 实现。
- **默认实现类**
  - `org.redisson.Redisson`
    - 静态工厂：`public static RedissonClient create(Config config)`
    - 典型用法：
      - 代码配置：
        - `Config config = new Config();`
        - `config.useSingleServer().setAddress("redis://127.0.0.1:6379");`
        - `RedissonClient client = Redisson.create(config);`
      - YAML 配置：
        - `Config config = Config.fromYAML("redisson.yml");`
        - `RedissonClient client = Redisson.create(config);`
    - 内部负责：
      - 解析 `Config`，根据部署模式创建相应 `ConnectionManager`。
      - 初始化命令执行服务 `CommandAsyncService` / `CommandSyncService`。
      - 封装各种 `R*` 接口的默认实现，如 `RedissonMap`、`RedissonLock` 等。

### 2.2 配置体系：`Config` 及各种模式配置

- **核心类**
  - `org.redisson.config.Config`
    - 全局配置根对象，持有以下模式之一：
      - `SingleServerConfig`
      - `ClusterServersConfig`
      - `SentinelServersConfig`
      - `MasterSlaveServersConfig`
      - `ReplicatedServersConfig`
      - `ProxyConfig` / `MultiClusterServersConfig` / `MultiSentinelServersConfig` 等。
    - 通用配置项：
      - 线程与事件循环：`threads`、`nettyThreads`
      - `codec`：序列化方式
      - `transportMode`：NIO / EPOLL 等
      - `lockWatchdogTimeout`：锁看门狗自动续期超时
      - 集群扫描间隔、重连策略、超时时间等。
- **单机模式示例：`SingleServerConfig`**
  - 典型字段：
    - `address`、`password`、`database`
    - `connectionPoolSize`、`connectionMinimumIdleSize`
    - `subscriptionConnectionPoolSize` 等。
- **集群/哨兵/主从配置**
  - `ClusterServersConfig`：slot 拓扑刷新、读写分离策略等。
  - `SentinelServersConfig`：哨兵地址、主名称、故障转移策略。
  - `MasterSlaveServersConfig`：手动指定主从节点，读写分离。

阅读路径：**先看 `Config`，再按你实际使用的部署模式打开对应 `*ServersConfig` 类，对照官方文档的 YAML 配置字段，很容易建立“配置项 → 行为”的整体认识。**

### 2.3 创建客户端的完整调用链（简化）

1. **用户侧**
   - 构造/加载 `Config`。
   - 调用 `Redisson.create(config)`。
2. **`Redisson.create` 内部**
   - 基于 `config` 判定部署模式：
     - `config.getSingleServerConfig() != null` → `SingleConnectionManager`
     - `getClusterServersConfig() != null` → `ClusterConnectionManager`
     - `getSentinelServersConfig() != null` → `SentinelConnectionManager`
     - ……
   - 构造对应的 `ConnectionManager` 实现，完成：
     - Redis 节点拓扑信息解析。
     - Netty 线程池与事件循环组初始化。
     - 命令编码/解码器、心跳、空闲检测等 handler 装配。
   - 基于 `ConnectionManager` 构造：
     - `CommandAsyncService`（异步命令执行核心）。
     - `CommandSyncService`（同步接口简单包装异步接口）。
   - 用以上组件创建 `Redisson` 实例并返回为 `RedissonClient`。
3. **完成后**
   - 客户端进入“就绪”状态，后续所有 `R*` 对象获取 & 调用，都会经由上述组件路由到底层 Redis 命令。

## 3. 底层网络与命令执行核心

### 3.1 Redis 原生客户端层：`org.redisson.client`

- **关键类与职责**
  - `RedisClient`
    - 封装单个 Redis 节点的连接工厂。
    - 负责创建 `RedisConnection`、连接复用、地址管理。
  - `RedisConnection`
    - 表示与 Redis 的一条物理连接（基于 Netty `Channel`）。
    - 提供 `send(CommandData<?>)` 等方法，将命令写入管道。
  - `RedisCommand` / `Protocol`
    - 命令元数据与协议封装，定义命令名、参数类型、返回值解析逻辑。
  - `CommandData` / `RedisExecutor`
    - 代表一次具体命令调用（包含回调、超时、重试信息）。
  - Netty handler 相关类：
    - 编码：将 `CommandData` 转为 RESP 协议字节流。
    - 解码：解析服务器响应，回填到 `CommandData` 的 `Promise`。

阅读建议：从 `RedisClient` → `RedisConnection` → 命令编码/解码 handler 的调用链，理解 **“一条 Redis 命令从 Java 代码到网络字节流再到响应回来的生命周期”**。

### 3.2 连接管理：`org.redisson.connection`

- **核心抽象**
  - `ConnectionManager`
    - 提供获取读/写连接、订阅连接等高层接口。
    - 屏蔽部署模式差异（单机/集群/哨兵/主从）。
  - 重要实现类：
    - `SingleConnectionManager`
    - `MasterSlaveConnectionManager`
    - `ClusterConnectionManager`
    - `SentinelConnectionManager`
    - `ReplicatedConnectionManager` 等。
- **主要职责**
  - **拓扑管理**
    - 集群模式：跟踪 slot → 节点映射，定期刷新。
    - 哨兵模式：监听主从切换，更新主节点地址。
  - **连接池**
    - 每个节点维护独立连接池（普通命令 + 订阅连接）。
    - 池内连接复用、空闲回收、最大/最小空闲连接控制。
  - **路由选择**
    - 根据命令类型与 key：
      - 写操作 → master。
      - 读操作根据 `ReadMode` 决定：master / slave / nearest。
      - 集群模式根据 slot 定位到具体节点。
  - **容错与重连**
    - 自动重连逻辑。
    - 节点宕机时的命令重试策略。

### 3.3 命令执行服务：`org.redisson.command`

- **核心类**
  - `CommandAsyncService`
    - Redisson 的“心脏”之一，封装了大部分发送命令逻辑。
    - 职责：
      - 基于 `ConnectionManager` 选择目标节点/连接。
      - 构造 `CommandData` 并发送到 `RedisConnection`。
      - 处理重试（如 `MOVED` / `ASK` 重定向、连接断开等）。
      - 处理命令超时与取消。
  - `CommandSyncService`
    - 基于 `CommandAsyncService` 的同步包装：
      - 发送异步命令 → 阻塞等待 `Future.get` → 返回结果。
  - `RedisExecutor`
    - 具体封装单次命令/批量命令的执行细节。

**理解方式**：选一个简单命令（如 `RBucket.get()`），在 IDE 中跟踪调用：

1. `RBucket.get()` → `RedissonBucket.get()`
2. `RedissonBucket.get()` → `CommandExecutor.read(...)`
3. `CommandExecutor` 实际多为 `CommandSyncService` / `CommandAsyncService` 的 façade。
4. 最终落到：
   - `CommandAsyncService.readAsync(...)` / `writeAsync(...)`
   - 选择 `ConnectionManager` 提供的连接。
   - 发往 `RedisConnection`，等待响应。

## 4. Redisson 针对 Redis 的核心封装一览

本节按“功能维度”梳理常用接口、默认实现类与底层 Redis 结构，并罗列主要源码入口，方便你按模块深入阅读。

### 4.1 分布式对象 & 简单数据结构（`org.redisson.api` / `org.redisson`）

- **对象 / 值：`RBucket`**
  - 接口：`org.redisson.api.RBucket<V>`
  - 实现：`org.redisson.RedissonBucket<V>`
  - 底层：`SET` / `GET` / `DEL` 等字符串操作。
  - 特点：
    - 通过 `Codec` 对任意 Java 对象序列化。
    - 支持 TTL、原子替换、比较并设置等。

- **计数与原子操作：`RAtomicLong`、`RAtomicDouble` 等**
  - 接口：`RAtomicLong`、`RAtomicDouble`、`RLongAdder`、`RDoubleAdder`
  - 实现：`RedissonAtomicLong` 等。
  - 底层：`INCRBY`、`DECRBY`、Lua 脚本保证复杂操作原子性。

- **BitSet：`RBitSet`**
  - 接口：`RBitSet`
  - 实现：`RedissonBitSet`
  - 底层：Redis `SETBIT` / `GETBIT` / `BITOP` / `BITCOUNT`。

### 4.2 集合与 Map（核心使用最广）

- **Hash：`RMap<K, V>`**
  - 接口：`org.redisson.api.RMap<K,V>`
  - 实现：`org.redisson.RedissonMap<K,V>`
  - 底层：`HSET` / `HGET` / `HDEL` / `HSCAN`。
  - 扩展：
    - `RMapCache<K,V>` / `RedissonMapCache<K,V>`：带 TTL 与淘汰策略的 Map。
    - `RLocalCachedMap<K,V>`：本地缓存 + Redis 回源。
  - 关键实现点：
    - 批量操作（如 `fastPut` 等）通过 pipeline / Lua 降低 RTT。
    - 迭代器封装 `HSCAN`，避免一次性加载全部数据。

- **List：`RList<V>`**
  - 接口：`RList<V>`
  - 实现：`RedissonList<V>`
  - 底层：`LPUSH` / `RPUSH` / `LINDEX` / `LRANGE` 等。

- **Set：`RSet<V>` / `RSortedSet<V>` / `RScoredSortedSet<V>`**
  - 对应 Redis `SET` / `ZSET` 结构。
  - 实现类：`RedissonSet<V>`、`RedissonSortedSet<V>`、`RedissonScoredSortedSet<V>` 等。

- **多值 Map：`RMultimap` 系列**
  - 接口：`RMultimap<K,V>`、`RListMultimap`、`RSetMultimap` 等。
  - 实现：`RedissonListMultimap`、`RedissonSetMultimap`。
  - 底层：将 key → collection 映射拆分为多个 Redis key 或特殊编码。

### 4.3 队列、消息与流

- **队列 / 阻塞队列**
  - 接口：
    - `RQueue<V>` / `RDeque<V>`
    - `RBlockingQueue<V>` / `RBlockingDeque<V>`
    - `RPriorityQueue<V>` / `RPriorityDeque<V>`
    - `RReliableQueue<V>`、`RTransferQueue<V>` 等。
  - 实现：`RedissonQueue`、`RedissonBlockingQueue` 等。
  - 底层：
    - 普通队列多基于 Redis `LIST`。
    - 优先级队列结合 `ZSET`。
    - 可靠队列会额外维护未确认队列、超时重投逻辑（Lua 脚本）。

- **延迟队列：`RDelayedQueue`**
  - 接口：`RDelayedQueue<V>`
  - 实现：`RedissonDelayedQueue`
  - 底层：
    - 使用 `ZSET` 存放“消息 + 触发时间戳”。
    - 后台任务扫描到期元素，移动到真实消费队列。
    - 大量使用 Lua 保证“从延迟集合转移到就绪队列”的原子性。

- **发布订阅：`RTopic` / `RPatternTopic`**
  - 接口：`RTopic`、`RPatternTopic`
  - 实现：`RedissonTopic`、`RedissonPatternTopic`
  - 底层：Redis `PUBLISH` / `SUBSCRIBE` / `PSUBSCRIBE`。
  - 特点：回调执行在线程池中，支持多种消息编解码。

- **流：`RStream`**
  - 接口：`RStream<K,V>`
  - 实现：`RedissonStream`
  - 底层：Redis Streams (`XADD` / `XREADGROUP` / `XACK` 等)。

### 4.4 分布式锁与同步器（核心中的核心）

- **普通锁：`RLock`**
  - 接口：`org.redisson.api.RLock`
  - 实现：`org.redisson.RedissonLock`
  - 底层：
    - 使用单个 key（如：`{name}:lock`）存储“锁持有者 + 重入计数”。
    - 加锁/解锁逻辑通过 Lua 脚本实现原子性：
      - 加锁：若 key 不存在则初始化；若持有者相同则重入计数 +1。
      - 解锁：重入计数减到 0 时删除 key；否则仅减计数。
    - 与 `lockWatchdogTimeout` 配合，后台看门狗线程定期执行 `PEXPIRE` 续期，避免长时间持有锁的线程在未显式解锁时被 Redis 过早删除。

- **公平锁：`RFairLock`**
  - 实现：`RedissonFairLock`
  - 底层：
    - 除主锁 key 外，还维护一个等待队列（list / zset）。
    - 通过 Lua 确保按进入队列顺序分配锁，避免饥饿。

- **读写锁：`RReadWriteLock`**
  - 接口：`RReadWriteLock`，内部包含 `RLock readLock()` / `RLock writeLock()`。
  - 实现：`RedissonReadWriteLock`、`RedissonReadLock`、`RedissonWriteLock`
  - 底层：
    - 使用 hash / string 记录读锁计数与写锁持有者。
    - Lua 逻辑确保“读读共享、读写互斥、写写互斥”。

- **红锁（多节点）：`RedissonRedLock`**
  - 接口：`RLock`（组合型）
  - 实现：`org.redisson.RedissonRedLock`
  - 底层：
    - 在多个独立 Redis 实例上分别加普通 `RLock`。
    - 基于 Redlock 算法：至少在多数节点上成功加锁才算成功。

- **信号量与计数器**
  - `RSemaphore` / `RedissonSemaphore`
    - 维护一个 permits 计数值。
    - `acquire` / `release` 使用 Lua 确保原子增减。
  - `RPermitExpirableSemaphore`：permit 带过期时间。
  - `RCountDownLatch` / `RedissonCountDownLatch`
    - 记录计数值，`countDown` 到 0 时唤醒等待线程。

**阅读建议**：任选一个锁（如 `RedissonLock`），顺着：

- 公共 API（`lock()` / `unlock()`）→
- 内部调用的 Lua 脚本组装与执行 → 
- 对应 Redis key 设计与语义

仔细理解一遍，后面阅读其他同步器就会非常顺畅。

### 4.5 限流、布隆过滤器等高级结构

- **限流器：`RRateLimiter`**
  - 接口：`RRateLimiter`
  - 实现：`RedissonRateLimiter`
  - 底层：
    - 使用 Redis key 保存速率配置与当前 token 数。
    - 核心操作（申请 token）通过 Lua 实现：
      - 根据当前时间与上次刷新时间计算应当生成的 token。
      - 判断是否可以扣减所需数量。
    - 支持多种模式：总体速率、每客户端速率等。

- **布隆过滤器：`RBloomFilter`**
  - 接口：`RBloomFilter<T>`
  - 实现：`RedissonBloomFilter<T>`
  - 底层：
    - 使用一个或多个 bitset（Redis `BIT*` 系列命令）。
    - 保存期望元素个数与误判率，根据这些参数生成多个 hash 函数种子。

## 5. 典型调用链示例：`RLock.lock()` 到 Redis 的全过程

以最典型的分布式锁为例，理清“从业务代码到 Redis 命令”的路径，能帮助你快速建立对整个框架的直觉。

1. **业务代码**
   - `RLock lock = redisson.getLock("order-lock");`
   - `lock.lock();`
2. **对象工厂**
   - `Redisson.getLock(name)`：
     - 构造 `RedissonLock`，注入：
       - `CommandAsyncService` / `CommandExecutor`
       - `ConnectionManager`
       - 锁名等元数据。
3. **锁实现逻辑**
   - `RedissonLock.lock()`：
     - 通过 Lua 脚本尝试设置锁 key（包含线程 ID、重入计数）。
     - 若失败，根据配置决定是否自旋重试/等待。
4. **命令执行**
   - `RedissonLock` 通过 `commandExecutor.evalWriteAsync(...)`：
     - 封装 Lua 脚本与参数为 `CommandData`。
     - 交由 `CommandAsyncService` 路由到合适节点。
5. **连接与网络**
   - `CommandAsyncService`：
     - `ConnectionManager` → 选写节点连接（单机/集群/哨兵下的 master）。
     - 通过 `RedisConnection.send()` 写入 RESP 流。
6. **响应处理**
   - Netty 读取服务器响应 → 解码 → 完成 `CommandData` 的 `Promise`。
   - `lock()` 同步接口最终等待该 Promise 完成，返回是否成功。
7. **看门狗续期**
   - 若锁设置了自动续期（默认启用）：
     - 后台 `LockWatchdog` 定期执行 Lua 更新 key 的 TTL。
     - 唯一持有者线程终止或显式 `unlock()` 时停止续期。

建议你在 IDE 中实际跟踪一次 `RLock.lock()` 调用链，对照上述步骤跳转到各个相关类，会对“启动流程 + 命令执行 + 功能封装”的整体联动有非常直观的感受。

## 6. 源码阅读路线建议（按优先级）

1. **入口与配置**
   - `org.redisson.api.RedissonClient`
   - `org.redisson.Redisson`
   - `org.redisson.config.Config` 及你实际使用的 `*ServersConfig`
2. **网络与命令执行**
   - `org.redisson.client.RedisClient` / `RedisConnection`
   - `org.redisson.connection.*ConnectionManager`
   - `org.redisson.command.CommandAsyncService` / `CommandSyncService`
3. **常用数据结构**
   - `RBucket` → `RedissonBucket`
   - `RMap` / `RMapCache` → `RedissonMap` / `RedissonMapCache`
   - `RList`、`RSet`、`RScoredSortedSet` 等常见集合。
4. **分布式同步原语**
   - `RLock` / `RedissonLock`
   - `RReadWriteLock` / `RedissonReadWriteLock`
   - `RFairLock` / `RedissonFairLock`
   - `RSemaphore`、`RCountDownLatch` 等。
5. **队列与消息**
   - `RQueue` / `RBlockingQueue` / `RDelayedQueue`
   - `RTopic` / `RPatternTopic`
   - `RStream`
6. **高级功能与集成（按需）**
   - `RRateLimiter`、`RBloomFilter`、`RHyperLogLog` 等。
   - JCache、Spring、Hibernate、MyBatis 集成模块中的适配实现。

## 7. 如何在本项目中进一步细化

- **结合实际使用场景标注“重点路径”**
  - 如果你在业务中主要用到了：
